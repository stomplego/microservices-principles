Вы работаете в крупной компании, которая строит систему на основе микросервисной архитектуры.
Вам как DevOps-специалисту необходимо выдвинуть предложение по организации инфраструктуры для разработки и эксплуатации.

## Задача 1: API Gateway 

Предложите решение для обеспечения реализации API Gateway. Составьте сравнительную таблицу возможностей различных программных решений. На основе таблицы сделайте выбор решения.

Решение должно соответствовать следующим требованиям:
- маршрутизация запросов к нужному сервису на основе конфигурации,
- возможность проверки аутентификационной информации в запросах,
- обеспечение терминации HTTPS.

Обоснуйте свой выбор.

#### Сравнительная таблица

| Критерий | NGINX | Traefik | Kong | Apache APISIX |
| :--- | :--- | :--- | :--- | :--- |
| **Маршрутизация** | ✅ Отличная (location, proxy_pass) | ✅ Отличная (авто-обнаружение) | ✅ Отличная (declarative config) | ✅ Отличная (динамическая) |
| **Аутентификация** | ✅ (auth_request + внешний сервис) | ✅ (Middleware: JWT/ForwardAuth) | ✅ Превосходная (богатые плагины) | ✅ Отличная (встроенные плагины) |
| **Терминация HTTPS** | ✅ Полная поддержка SSL/TLS | ✅ Автоматическая (Let's Encrypt) | ✅ Поддерживается | ✅ Поддерживается (HTTP/3) |
| **Простота настройки** | Средняя (ручные конфиги) | **Высокая** (авто-discovery) | Средняя (требует БД) | Сложная (требует etcd) |
| **Хранение конфигурации** | Файлы на диске | Файлы / провайдеры | База данных (Postgres) | etcd (кластер) |
| **Производительность** | **Очень высокая** (C) | Высокая (Go) | Высокая (Nginx + Lua) | Очень высокая (Nginx + LuaJIT) |
| **Зависимости** | **Нет** | **Нет** | Есть (БД) | Есть (etcd) |
| **Эксплуатация** | Простая (один бинарный файл) | Простая | Сложная (требует администрирования БД) | Сложная (etcd) |

Выбор сделан в пользу **NGINX** по следующим причинам:
- Соответствие требованиям. nginx из коробки решает все три задачи.
- Простота в условиях задачи. Работа ведётся в линукс, nginx устанавливается через штатный apt и не требует дополнительных сервисов, в отличие от kong или apisix.
- Вся конфигурация хранится в текстовых файлах, что позволяет удобно версионировать её в Git.
- Надёжность и производительность. nginx - эталонный веб-сервер с высокой пропускной способностью и низким потреблением памяти.

## Задача 2: Брокер сообщений

Составьте таблицу возможностей различных брокеров сообщений. На основе таблицы сделайте обоснованный выбор решения.

Решение должно соответствовать следующим требованиям:
- поддержка кластеризации для обеспечения надёжности,
- хранение сообщений на диске в процессе доставки,
- высокая скорость работы,
- поддержка различных форматов сообщений,
- разделение прав доступа к различным потокам сообщений,
- простота эксплуатации.

Обоснуйте свой выбор.

#### Сравнительная таблица

| Критерий | Apache Kafka | RabbitMQ | Redis Pub/Sub | NATS |
| :--- | :--- | :--- | :--- | :--- |
| **Кластеризация** | ✅ Отличная (встроенная) | ✅ Хорошая (кластеризация Erlang) | ❌ Слабая (Sentinel/Cluster для данных) | ✅ Хорошая (Supercluster) |
| **Хранение на диске** | ✅ Да (по умолчанию) | ✅ Да | ❌ Нет (в памяти, RDB/AOF не для сообщений) | ❌ Нет (JetStream добавляет, но опционально) |
| **Скорость работы** | Высокая (до 100k msg/s) | Средняя (до 50k msg/s) | **Очень высокая** (в памяти) | **Очень высокая** (лёгкий протокол) |
| **Форматы сообщений** | ✅ Любые (бинарные) | ✅ Любые | ✅ Любые | ✅ Любые |
| **Разделение прав (ACL)** | ✅ Да (PLAIN, SCRAM, Authorizer) | ✅ Да (virtual hosts, permissions) | ❌ Слабое (только пароль) | ❌ Слабое (базовое) |
| **Простота эксплуатации** | Сложная (требует ZooKeeper/KRaft) | **Простая** (не требует доп. сервисов) | Простая | Простая |
| **Гарантии доставки** | At-least-once, Exactly-once | At-least-once | At-most-once | At-least-once |
| **Ретенция сообщений** | ✅ По времени/объёму | ❌ (очередь удаляет после прочтения) | ❌ | ❌ (через JetStream) |

Для решения выбран **Apache Kafka** по следующим причинам:
- Соответствие ключевым требованиям
- Преимущества для микросервисной архитектуры (поддержка событийно-ориентированной архитектуры, возможность воспроизводить события благодаря хранению сообщений, 
масштабирование путём увеличения партиций)
- Компромисс по простоте эксплуатации. Хотя kafka требует zookeeper, это плата за надёжность и производительность. Для RabbitMQ эксплуатация проще, но он хуже 
справляется с большими объёмами данных и ретенцией
- Форматы сообщений. Kafka не навязывает схемы, поддерживает любые форматы (JSON, Avro, Protobuf, бинарные данные)
